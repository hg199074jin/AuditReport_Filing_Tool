## 1. 背景与动机

### 1.1 现状

用户目前用**两个独立软件**完成审计报告报备工作,它们是同一个业务流程的前后两步:

| 软件 | 位置 | 作用 |
|---|---|---|
| 审计报备-上传 | `审计报备-上传\` | 给报告加盖 CPA 章 + 事务所章、拼接被审计单位盖章的报表/附注,生成上传注协系统的报告 |
| 审计报告-下载 | `审计报告-下载\` | 把注协赋码后的报告里盖章页替换为无章版、保留二维码,生成可打印的最终报告 |

完整业务流程:

```
原始审计报告.pdf (事务所内部版)
        │
        ▼  [步骤①·上传软件]
   加盖CPA/事务所章 + 拼接盖章报表/附注
        │
        ▼
   上传报告.pdf ──────► 注协系统赋码(人工操作,产生流程断点)
        │
        ▼  [步骤③·下载软件]
   替换盖章页为无章版 + 保留二维码
        │
        ▼
   可打印报告.pdf
```

### 1.2 痛点(用户确认)

1. **参数重复填写**:签字页页码、报表页范围等信息,两个软件都要各填一遍,重复且易错。
2. **割裂感强**:两个软件界面/操作习惯不同,要在两套心智模型间切换。
3. **流程不顺**:要回忆"先开哪个、生成什么、再喂给哪个",隔段时间易忘。

### 1.3 核心洞察

- 同一批页码(签字页、报表页等)在两个软件里是**同一组信息**,只是用途不同——上传时盖章/拼接,下载时替换。
- 处理的是**同一批文件**(原始报告贯穿两步)。
- **一直由同一人从头用到尾**(用户确认),适合做成连贯流程。

## 2. 设计决策

### 2.1 选定方案:步骤状态 + 侧面步骤导航栏

从三个候选方案中选定:

| 方案 | 说明 | 取舍 |
|---|---|---|
| ~~方案一:统一入口+参数一次配置~~ | 公共参数填一次,两步可独立点 | 基础,够用但缺状态管理 |
| **方案二:彻底一键流+状态记忆** | 侧边栏步骤导航,体现进度 | **选定** |
| ~~方案三:两软件+共享配置~~ | 共享json同步参数 | 治标不治本,割裂感仅减轻 |

用户进一步选择**侧面导航栏按"步骤(流程)"组织**(而非按报告列表),体现线性进度。

### 2.2 关键决策汇总

| 维度 | 决定 | 来源 |
|---|---|---|
| 状态持久化 | **不做跨次持久化**,本次运行内记忆即可 | 用户确认赋码很快,连续做完 |
| 印章路径 | **跨次记忆**(写配置文件) | 印章固定不变,每次重选很烦 |
| 报告路径/页码 | 本次内记忆,不跨次 | 每份报告都不同 |
| 签字页处理 | 步骤①盖章 + 步骤③替换(两步都要) | 用户确认 |
| 状态切换 | 线性约束(①→②→③),重做①清空②③ | 用户确认合理 |

### 3.2 侧边栏三步

| 步骤 | 标识 | 内容 |
|---|---|---|
| ① 准备上传版 | ✓(完成)/●(进行中)/○(未到) | 配置印章、盖章报表/附注 → 生成上传报告 |
| ② 等待赋码 | 提示状态 | 显示"已生成上传报告,请去注协赋码后回来做步骤③"+ "打开所在文件夹"按钮 |
| ③ 处理赋码版 | 处理状态 | 选赋码报告 → 替换盖章页+保留二维码 → 生成可打印报告 |

### 3.3 公共配置区(始终可见)

| 参数 | 归属 | 说明 |
|---|---|---|
| 原始报告路径 | **公共** | ①作盖章基底,③提供无章替换页 |
| 盖章页页码 | **公共** | ①盖章定位(对应上传项目 `seal_page`,审计报告里通常与签字页同页,如第5页);③作为签字页替换目标 |
| 报表页范围 | **公共** | ①拼接盖章报表的范围参考;③转换为 `replace_rules` 中的报表替换规则 |

> **注意**:下载项目③的替换逻辑依赖的是 **`replace_rules` 列表**(注协起止页↔原始起止页的逐页映射),而非单一"页码"。公共配置里的"盖章页""报表页范围"在步骤③启动时,需转换为对应的 replace_rules 条目(盖章页→单页替换规则;报表范围→区间替换规则),并按 config.json 的 `auto_add_last_page_rule` 追加"最后一页替换"规则(原始报告末页↔赋码报告末页)。附注末页特指"自动追加的最后一页替换规则",非单独配置项。

### 3.4 各步骤专属配置

| 参数 | 步骤 | 说明 |
|---|---|---|
| CPA章图片 | ① | 跨次记忆(配置文件,见 3.5) |
| 事务所章图片 | ① | 跨次记忆(配置文件,见 3.5) |
| 盖章报表PDF | ① | 本次内 |
| 盖章附注PDF | ① | 本次内 |
| 赋码报告路径 | ③ | 本次内 |

### 3.5 印章库与"上次选用记忆"的区别

需要区分两个概念(原上传项目只有前者):

| 概念 | 说明 | 持久化 |
|---|---|---|
| **印章库管理** | 维护多枚印章的命名→路径映射(如"大梁公章""CPA潘利"),供每次选用。复用上传项目 `ConfigManager` + `seals_config.json` | 已有,复用 |
| **上次选用记忆** | 记录"上次用的是哪两枚印章(CPA/事务所)",下次打开自动选中。**原项目没有此功能,需新增** `last_used` 字段 | **新增** |

即:`seals_config.json` 新增 `last_used_cpa` 和 `last_used_firm` 两个字段,存上次选用的印章路径。打开软件时自动选中这两枚,用户可改选印章库里的其他枚。

## 4. 处理逻辑

### 4.1 步骤①准备上传版

**复用"审计报备-上传"项目逻辑。**

**输入**:
- 原始报告(公共)
- CPA章、事务所章图片
- 盖章报表PDF、盖章附注PDF

**处理**:
- 给签字页加盖 CPA 章 + 事务所章。复用上传项目的**完整盖章链路**(任一缺失都无法工作):
  - `find_seal_page`:模糊匹配标记句定位盖章页
  - `find_seal_anchors`:提取事务所名、"盖章"、CPA 签字位等锚点
  - `compute_seal_positions`:按物理尺寸(cm→pt 换算)、CPA y 坐标排序计算印章位置
  - `_get_seal_target_size`、`_firm_name_center_x`:尺寸计算、事务所名中心定位
  - `add_seals_to_page`:实际盖章(依赖前述算出的 `seal_positions` 和预览渲染图 `page_image` 做坐标换算)
- 把盖章报表/附注拼接进报告(复用上传项目的拼接逻辑)

**输出**: `上传报告_<原始报告文件名>.pdf`(沿用原项目"前缀+原文件名"命名方式,不提取被审计单位名——原项目无此逻辑,提取规则未定义故不引入)

**成功后**: 侧边栏①变✓,自动跳到步骤②状态。

### 4.2 步骤②等待赋码

**纯提示页,不执行任何处理代码。**

界面内容:
- 显示已生成的上传报告路径
- "打开所在文件夹"按钮:用 `os.startfile(os.path.dirname(path))`(Windows)打开资源管理器;若路径失效(目录已删/移动),捕获异常并提示"输出目录不存在,请检查"
- 操作指引:1.登录注协上传 2.等赋码 3.下载赋码报告 4.回这里点步骤③

### 4.3 步骤③处理赋码版

**复用"审计报告-下载"项目逻辑。**

**输入**:
- 原始报告(公共,提供无章替换页)
- 赋码报告(本步专属)

**处理**:
- 用公共页码配置(签字页/报表页/附注末页),把赋码报告里的盖章页替换为原始报告的无章版
- 替换页自动抠取并贴回二维码(复用下载项目的保码逻辑:`_extract_qr_strip` 像素分析定位方块 + `_paste_qr_strip` 横竖版适配)
- 横版页按页宽等比缩放二维码

> **页码映射说明**:`replace_rules` 是"注协页 ↔ 原始页"的双向映射。公共配置只提供**原始报告侧**页码(如盖章页=原始第5页),**赋码报告侧**对应页码需用户在③界面确认/输入(沿用原下载项目做法——原项目本就由用户手填注协侧页码)。可默认填入与原始侧相同的页码(标准报告两侧页码通常一致),用户核对后可改。

**输出**: `可打印报告_<赋码报告文件名>.pdf`(沿用原下载项目"前缀+原文件名"命名方式)

**成功后**: 侧边栏③变✓,显示完成提示。

### 4.4 预览区

随当前步骤显示对应 PDF:
- 步骤①:预览原始报告(确认页码配置)
- 步骤③:预览赋码报告(确认替换范围)

## 5. 状态流转

### 5.1 状态机

```
                 生成上传报告成功
    ①准备 ──────────────────────► ②赋码
   (进行中)                         (等待)
     ▲                                │
     │ 用户重做                        │ 用户回来,选赋码报告
     │                                ▼
     └─────────────────────────── ③下载
                                   (进行中)
                                     │
                                     │ 生成可打印报告成功
                                     ▼
                                    完成
```

### 5.2 状态规则

- **初始**: ①进行中,②③灰色不可点
- **①成功后**: ①✓,②可点,③仍灰(还没赋码报告)
- **进入③后**: ①②都✓,③进行中
- **③成功后**: 全部✓
- **重做①**: 清空②③状态(换报告意味着旧进度失效)。输出文件采用**覆盖同名**策略(沿用原项目行为):若上次生成的"上传报告_xxx.pdf"已传给注协,用户应自行改名备份;重做①覆盖的是工作区的中间产物,不影响已上传的文件。若目标文件正被占用,走第6节"输出文件被占用"的友好提示。

### 5.3 记忆范围

- **本次运行内有效**(内存):所有参数和状态
- **跨次持久化**(配置文件):仅印章库配置 + 上次选用印章路径(见 3.5)
- 关闭软件再打开 → 报告路径/页码回到初始状态(页码用 config.json 默认值预填),印章自动选中上次的

## 6. 错误处理

统一应用两个项目已改进的友好中文提示:

| 错误场景 | 提示内容 |
|---|---|
| 输出文件被占用 | "请关闭打开该文件的程序(PDF阅读器/WPS/浏览器等)" |
| 文件不存在 | "请检查路径是否正确、文件是否存在" |
| 页码超范围 | "注协PDF最多N页,原PDF最多N页" |
| 缺少必要文件 | 提示具体缺哪个文件 |
| 输出目录打开失败 | "输出目录不存在,请检查"(步骤②"打开所在文件夹"按钮的路径失效场景) |

每个步骤的"生成"按钮点击后,先校验该步骤所需文件齐全,不齐则提示缺哪个,不进入处理。

处理过程中复用现有进度条对话框(百分比 + 当前操作描述)。

### 7.2 模块职责

| 模块 | 职责 | 来源 |
|---|---|---|
| `main.py` | 仅启动入口(转调 `gui.py` 的 main 函数,无独立逻辑,沿用原项目两行式写法) | 复用 |
| `gui.py` | 步骤导航界面、公共配置区、预览区、状态流转编排 | **新写** |
| `step_upload.py` | 盖章、拼接报表/附注,生成上传报告 | 从"上传"项目提炼为函数 |
| `step_download.py` | 替换盖章页、抠码贴回,生成可打印报告 | 从"下载"项目提炼为函数 |
| `pdf_common.py` | PDF渲染、坐标转换等两步共用底层操作 | 从两项目提取公共部分 |
| `config_manager.py` | 印章库读写 + **上次选用印章记忆**(新增 last_used 字段)、默认页码读写 | 复用上传项目 + 扩展 |

### 7.3 复用原则

不复制粘贴大段代码,而是把现有逻辑**提炼成函数**:
- 上传项目**完整盖章链路** → `step_upload.py`:`find_seal_page` → `find_seal_anchors` → `compute_seal_positions`(`_get_seal_target_size`、`_firm_name_center_x`)→ `add_seals_to_page`;以及报表拼接逻辑
- 下载项目 `_extract_qr_strip`(抠码)、`_paste_qr_strip`(贴回)、`process_files`(替换 + replace_rules/insert_rules)→ `step_download.py`
- 两项目共用的 fitz 渲染、页码转换 → `pdf_common.py`

`gui.py` 只管界面和流程编排,处理逻辑下沉到 step 模块。每个模块单一职责,可独立理解和测试。

## 8. 依赖与配置

### 8.1 依赖

`requirements.txt`:
```
PyMuPDF
Pillow
numpy
```

**说明**:`numpy` 是下载项目步骤③抠码逻辑(`_extract_qr_strip` 用 `np.array` 做像素列分布统计定位二维码方块)的必需依赖。原下载项目的 `requirements.txt` 漏写了 numpy(原项目已存在的隐患),本整合项目必须补上,否则步骤③会因 `import numpy` 失败而崩溃。

> **关于 PyPDF2**:上传项目 `pdf_processor.py` 顶部虽 `import PyPDF2` 作 fallback,但主路径已全切 fitz,整合项目提炼代码时直接去掉 PyPDF2 fallback,故不列入依赖。

### 8.2 配置文件

- `seals_config.json`:印章尺寸(公章4×4cm、CPA 3.4×2.3cm)、定位偏移常量等(复用上传项目已调好的参数)
- `config.json`:默认页码(签字页5、报表6-11等),打开即预填

### 8.3 启动

- `审计报告报备工具.bat`:Windows 启动脚本(沿用现有项目方式)
- 桌面快捷方式(可选)

## 9. 已知约束与风险

### 9.1 流程断点不可避免

步骤②③之间,用户必须在注协网站手动操作(上传→赋码→下载),软件无法自动化。步骤②作为过渡提示页处理此断点。

### 9.2 签名校验提示

步骤③输出的 PDF 在阅读器里可能提示"签名已修改/无效"(注协数字签名机制决定),但二维码正常显示、打印正常。用户已确认可接受。

### 9.3 不做跨次持久化的取舍

- **好处**:不引入文件读写、版本冲突、路径失效的复杂度(YAGNI)
- **代价**:关闭软件后,报告路径和页码配置丢失,需重填
- **缓解**:印章路径跨次记忆(最常变的反而省了),页码有默认配置预填

## 10. 不在本次范围(YAGNI)

以下功能明确**不做**,避免过度设计:

- 项目文件持久化(保存/打开 .project 文件)
- 多报告批量处理队列
- 报告历史记录
- 注协系统自动登录/上传(技术上不可行,且涉及账号安全)
- 云同步、多机协作

> **持久化边界澄清**:本设计唯一的持久化是「印章库 + 上次选用印章」(写入 `seals_config.json`),因为印章固定不变、每次重选很烦。报告路径、页码等参数**均不持久化**,关闭即丢失(页码用 config.json 默认值预填)。"不做项目文件持久化"指的就是后者,与印章记忆不冲突。

如未来需要,可按需增量添加。
